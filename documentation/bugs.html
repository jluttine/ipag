Known bugs

Out of memory situations are not taken into account.

Memory leaks

We have used valgrind to detect memory leaks. If we initialize SDL and quit immediately without loading any of our own classes, there are about a thousand missing frees. If we play the game for a while and quit, there are about 20.000 news and about a thousand missing frees, actually a bit fewer than if we only load SDL and quit. It's impossible to say if there are any memory leaks of our own. Valgrind gives a few errors when the game starts but they all concern SDL and there is nothing we can do about it.

What could have been done better

The current architecture is a bit messy and it doesn't allow us to add multiplayer network game. Graphics and sound should be separated better from game functionality and we should consider some kind of a client/server architecture to make network play possible. If we would have included network play from the start, it would have forced us to design the architecture better. The feature was seriously considered but it was dropped because of the tight schedule.

Projectiles were vey problematic. Players have lots of similar weapons but we don't want to have a unique object for each one. We chose to have GameEngine keep one instance of each projectile type in store and clone it when necessary. This isn't too elegant but it works pretty well. There could be a switch which would create a new object of selected type but that wouldn't be scalable. Think of loading the projectile types from a resource file. A better projectile system would be welcome but it is difficult to come up with one.

All game features were kept in and the game turned out exactly as we planned. Destructible planets might be a nice feature, but it would bring some problems too. Graphics can always be improved. There could be lots of different explosion types, and an efficient particle engine would allow us to add exhaust gases for missiles. There could be fonts of different sizes and colored text, not to mention cool buttons.
